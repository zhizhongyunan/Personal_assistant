# 第三周面试题整理 - RAG 核心技术（一）

## 🔹 一、基础题（必会）

### Q1: RAG 是什么？解决了什么问题？

**参考答案：**

RAG = **Retrieval-Augmented Generation**（检索增强生成）

**解决的问题：**
1. **知识截止**：LLM 知识固定在训练日期，无法获取最新信息
2. **私有数据**：LLM 无法访问企业内部数据
3. **幻觉问题**：LLM 容易胡说八道，RAG 提供事实依据

**工作流程：**
```
用户提问 → 检索相关文档 → 文档 + 问题一起给 LLM → LLM 基于文档回答
```

---

### Q2: 向量数据库的作用是什么？

**参考答案：**

向量数据库用于**存储和检索向量**，支持语义搜索。

**传统数据库 vs 向量数据库：**
| 对比项 | 传统数据库 | 向量数据库 |
|--------|------------|------------|
| 存储内容 | 文本/数字 | 向量（数字列表） |
| 检索方式 | 精确匹配 | 语义相似度 |
| 示例 | MySQL, PostgreSQL | Chroma, Pinecone, pgvector |

**核心功能：**
- 存储：文本 → Embedding → 向量
- 检索：查询 → 向量 → 找最相似的 N 个

---

### Q3: 为什么需要文本分块（Chunking）？

**参考答案：**

**原因：**
1. **LLM 上下文限制**：一次只能处理有限长度
2. **检索精度**：小块更容易精确匹配
3. **成本优化**：只检索相关块，节省 token

**分块策略：**
- 太大 → 检索不精确，浪费 token
- 太小 → 丢失上下文，语义不完整

---

### Q4: chunk_overlap 的作用是什么？

**参考答案：**

`chunk_overlap` 是**块之间的重叠部分**，用于保持上下文连贯。

**示例：**
```
原文：A B C D E F G H I

无重叠 (overlap=0)：
块 1: A B C
块 2: D E F
块 3: G H I

有重叠 (overlap=1)：
块 1: A B C
块 2: C D E  ← C 重叠
块 3: E F G  ← E 重叠
块 4: G H I  ← G 重叠
```

**作用：**避免关键信息被切分到两块之间导致语义丢失。

---

### Q5: Embedding 的工作原理？

**参考答案：**

Embedding 是**把文本转成向量**的过程。

**原理：**
```
文本 → Embedding 模型 → 向量（768/1536 维）

训练目标：语义相近的文本，向量距离近
```

**距离计算：**
- 余弦相似度：cos(θ) = (A·B) / (|A|×|B|)
- 结果越接近 1，语义越相似

## 🔹 二、中阶题（拉开差距）

### Q6: Chroma 和传统数据库的区别？

**参考答案：**

| 对比项 | Chroma（向量数据库） | MySQL（传统数据库） |
|--------|---------------------|---------------------|
| 存储 | 向量 + 元数据 | 表格数据 |
| 检索 | 语义相似度 | SQL 精确匹配 |
| 查询 | `similarity_search("AI")` | `SELECT * WHERE keyword='AI'` |
| 适用 | 语义搜索 | 事务处理 |

**代码对比：**
```python
# Chroma - 语义搜索
results = db.similarity_search("AI 框架", k=2)

# MySQL - 关键词匹配
results = db.query("SELECT * WHERE content LIKE '%AI%'")
```

### Q7: 如何选择合适的 Embedding 模型？

**参考答案：**

**选择因素：**

| 因素 | 说明 | 推荐 |
|------|------|------|
| 语言 | 中文/英文/多语言 | 中文选 bge/m3e，英文选 OpenAI |
| 成本 | 付费/免费 | 预算充足选 OpenAI，否则选开源 |
| 精度 | 维度越高越精确 | 通用场景 768 维足够 |
| 部署 | 云端/本地 | 离线场景选本地模型 |

**2026 推荐：**
- 首选：`text-embedding-3-small`（性价比高）
- 中文免费：`bge-large-zh` 或 `m3e-base`
- 高精度：`text-embedding-3-large`

---

### Q8: 分块大小如何选择？太大/太小有什么问题？

**参考答案：**

**推荐范围：**
- 短文本（邮件/消息）：200-300 字符
- 中文本（文章）：400-600 字符
- 长文本（论文/书籍）：800-1000 字符

**问题对比：**
| 问题 | 太大 | 太小 |
|------|------|------|
| 检索精度 | ❌ 不精确 | ✅ 精确 |
| 上下文 | ✅ 完整 | ❌ 丢失 |
| Token 消耗 | ❌ 多 | ✅ 少 |
| 语义理解 | ❌ 混杂 | ❌ 碎片化 |

### Q9: RAG 系统中，检索失败如何优化？

**参考答案：**

**优化策略：**

1. **调整分块大小**
   ```python
   splitter = RecursiveCharacterTextSplitter(
       chunk_size=300,  # 改小/改大
       chunk_overlap=30
   )
   ```

2. **调整检索数量**
   ```python
   results = db.similarity_search(query, k=5)  # 增加 k
   ```

3. **使用混合检索**
   ```python
   # 向量检索 + 关键词检索
   retriever = EnsembleRetriever(
       vector_retriever,
       keyword_retriever
   )
   ```

4. **添加 Rerank**
   ```python
   # 先检索 50 个，再精排到 5 个
   results = reranker.rank(results[:50], top_k=5)
   ```

### Q10: 如何处理长文档的 RAG？

**参考答案：**

**策略：**

1. **分层检索（Hierarchical Retrieval）**
   ```
   文档 → 章节 → 段落 → 句子
   先检索章节，再检索段落
   ```

2. **Parent Document Retriever**
   ```python
   # 检索小块，返回大块（包含父文档上下文）
   retriever = ParentDocumentRetriever(
       child_chunk_size=200,
       parent_chunk_size=800
   )
   ```

3. **摘要检索**
   ```
   先检索文档摘要，确定相关文档
   再检索详细内容
   ```
